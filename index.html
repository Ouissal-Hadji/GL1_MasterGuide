📚 Chapitre 1 — Introduction au Génie Logiciel
Définitions, qualités, crise du logiciel, processus de développement
💡

Définition du Logiciel

Logiciel = Programmes + Données + Documents

Le Génie Logiciel (GL) = ensemble de méthodes, techniques et outils pour la conception, développement et maintenance de logiciels, en appliquant des principes d'ingénierie classiques.

🎯

Objectifs du GL

  • Optimiser coût/délai de développement
  • Satisfaire les besoins client
  • Assurer haute qualité (performance, fiabilité, sécurité)
  • Faciliter maintenance et évolution

⚠️

Crise du Logiciel

  • Retards dans les délais
  • Dépassement de budget
  • Logiciels non fiables
  • Difficiles à maintenir
  • Besoins non satisfaits
🔄

Modèles de Développement

  • Cascade : phases séquentielles
  • V : tests à chaque phase
  • Spirale : itératif + risques
  • Agile : sprints, incrémental
  • UP/RUP : unifié itératif
🌍

Environnement du Logiciel

  • Utilisateurs : humains (médecins, clients…)
  • Autres logiciels : SGBD, API, OS
  • Matériel : capteurs, écrans, CPU
🔷 Chapitre 2 — Modélisation Orientée Objet & UML
Paradigme OO, présentation UML, types de diagrammes
🧩

Paradigme Orienté Objet

  • Objet = identité + état + comportement
  • Classe = regroupement d'objets similaires
  • Encapsulation : masquer les détails
  • Héritage : réutiliser les propriétés
  • Polymorphisme : même méthode, comportements différents
📐

UML en bref

UML = Unified Modeling Language — Langage visuel standard pour spécifier, construire et documenter un système logiciel [OMG, 1997].

  • UML 1 → 9 diagrammes
  • UML 2.5 → 14 diagrammes
  • Pas une méthodologie, un langage

Types d'Acteurs

Acteur Principal : déclenche/initie les CUs, bénéficiaire direct.
Ex : Client dans un DAB

Acteur Secondaire : aide à fournir le service, maintenance.
Ex : Banque dans un DAB

Acteur Système : représenté par un rectangle stéréotypé «actor»

🔗

Relations CU ↔ CU

IncludeX toujours inclut Y (obligatoire, factorisation)
ExtendX peut étendre Y (optionnel, conditionnel)
GénéralisationX est cas particulier de Y (spécialisation)
💡 Mnémonique : Include = inclusion obligatoire (flèche pointillée de X vers Y). Extend = extension optionnelle (flèche pointillée de X vers Y, sens inverse).

🏛️ Anatomie d'une Classe UML — Examen Q1
Extrait de l'examen Mai 2025 : code Java → DCL. Classes abstraites, héritage, attributs statiques.
public abstract class Product { protected String Ref; public Date launch; abstract float CalculValue(int state); } public class Phone extends Product { private boolean esim; static int tax = 25; @Override float CalculValue(int state){...} }
🔑 Règle : Attribut/méthode souligné = statique. Nom en italique ou stéréotype «abstract» = abstrait. Visibilité : + public, # protégé, - privé.
📋 Syntaxe complète des attributs et opérations

Attribut :

<visib> [/] <nom> : <type> [multiplicité] = <val init>
  • + public
  • # protégé
  • - privé
  • / dérivé (calculé)
  • u souligné = statique

Opération :

<visib> <nom>(params): <type retour>
  • in paramètre lecture seule (défaut)
  • out valeur retournée en param
  • inout modifiable

Q1 2.5 pts Diagramme de Classes depuis code Java
Donnez le DCL UML correspondant au code Java suivant. Représentez correctement les visibilités, l'abstraction, l'héritage et les attributs statiques.

public abstract class Vehicle { protected String brand; public int year; abstract double calculateEmission(int km); } public class ElectricCar extends Vehicle { private int batteryKWh; static double emissionFactor = 0.04; @Override double calculateEmission(int km) { /*...*/ } }
Barème : 1pt/classe + 0.5pt héritage − 0.5pt/erreur
Corrigé :
  • «abstract» Vehicle (nom en italique ou stéréotype): # brand: String, + year: int, méthode «abstract» calculateEmission(km:int): double
  • ElectricCar hérite de Vehicle (flèche triangle vers Vehicle): - batteryKWh: int, emissionFactor: double = 0.04 (souligné car statique), calculateEmission(km:int): double
  • Héritage = flèche avec triangle vide pointant vers Vehicle
  • Erreurs fréquentes : oublier l'underline pour statique, visibilité incorrecte, oublier le stéréotype abstract
Q2 3.5 pts DCL avec classe association et agrégation
Donnez un DCL modélisant le système suivant :
  • Un Employé (matricule, nom, dateEmbauche) peut participer à plusieurs Projets (codeP, titre, budget)
  • Sa participation enregistre son rôle et le nbHeures travaillées
  • Un Département emploie de 3 à 20 Employés ; un Employé appartient à exactement 1 Département
  • Supprimer la classe association en gardant un DCL équivalent
Barème : 4 × 0.5pt/classe avec attributs + 3 × 0.5pt/association avec cardinalité
Corrigé :
  • Département : nomDept, localisation → agrégation ◇ vers Employé avec 1 → 3..20
  • Employé : matricule (souligné = clé), nom, dateEmbauche → association vers classe Affectation
  • Affectation (ex-classe association) : rôle: String, nbHeures: int → avec * ← 1 vers Employé et * ← 1 vers Projet
  • Projet : codeP, titre, budget
  • Cardinalités : Employe 1 ← * Affectation * → 1 Projet + Département ◇ 1 → 3..20 Employe
Q3 3 pts Reconstruire un DCL depuis une BD relationnelle
Reconstruisez le DCL à partir des tables relationnelles suivantes :
Etudiant (numE, nomE, dateNaiss, #numFiliere) Filiere (numFiliere, nomFiliere, faculte) Module (codeM, intitule, credits) Inscription (numE, codeM, annee, note)
Barème : 4 × 0.5pt/classe + 2 × 0.5pt/association
Corrigé :
  • Filiere : numFiliere, nomFiliere, faculte
  • Etudiant : numE, nomE, dateNaiss → association 1 ← * avec Filière (#numFiliere = FK)
  • Inscription (classe association car clé composée + attributs propres) : annee, note → * ← Etudiant et * → Module
  • Module : codeM, intitule, credits
  • Relations : Filiere 1 → * Etudiant | Etudiant * ↔ * Module via Inscription(annee, note)
Q4 5 pts Diagramme de Cas d'Utilisation — sans héritage
Donnez un DCU équivalent au suivant, sans liens d'héritage entre acteurs ni entre CUs :

Système de gestion d'une bibliothèque :
  • Lecteur hérite de Utilisateur
  • Bibliothécaire hérite de Utilisateur
  • CU Gérer Livres est généralisé en : Ajouter Livre, Modifier Livre, Supprimer Livre
  • Utilisateur peut S'authentifier
  • Lecteur peut : Rechercher Livre, Emprunter Livre, Retourner Livre
  • Bibliothécaire peut : Ajouter Livre, Modifier Livre, Supprimer Livre, Imprimer Catalogue (optionnel)
  • Chaque CU de gestion «include» S'authentifier. Lecteur «include» S'authentifier aussi.
  • Imprimer Catalogue «extend» Rechercher Livre
Barème : Supprimer héritage acteurs (0.5), Supprimer CU Gérer Livres (0.5), Liens Lecteur/CUs (1), Liens Bibliothécaire/CUs (1), includes → S'authentifier (1), extend Imprimer (1)
Corrigé — Transformations clés :
  • Supprimer CU "Gérer Livres" + l'héritage entre CUs → garder les 3 CUs spécifiques directement
  • Supprimer héritage Lecteur↑Utilisateur et Bibliothécaire↑Utilisateur
  • Lecteur → relier directement à : Rechercher Livre, Emprunter Livre, Retourner Livre, S'authentifier
  • Bibliothécaire → relier directement à : Ajouter, Modifier, Supprimer Livre + S'authentifier
  • Ajouter Livre «include» S'authentifier, Modifier Livre «include» S'authentifier, Supprimer Livre «include» S'authentifier
  • Imprimer Catalogue «extend» Rechercher Livre (flèche pointillée de Imprimer vers Rechercher)
Q5 2 pts Diagramme de Séquence avec fragments combinés
Modifiez le DSE ci-dessous pour que :
  • Les messages m1 et m2 s'exécutent simultanément (parallèle)
  • m3 soit synchrone et s'exécute seulement si N > 0
  • Si erreur = true alors m4 s'exécute et le reste est abandonné
  • Un message m5 crée un nouvel objet de Logger
(Acteurs : :Client → Obj1:Service, puis m5 crée :Logger)

Barème : 3 × 0.5pt/fragment + 0.5pt création objet
Corrigé :
  • PAR : cadre avec 2 opérandes séparés par tirets. Opérande 1 = m1 (flèche Client→Service). Opérande 2 = m2 (flèche Client→Service). Les deux s'exécutent en même temps.
  • OPT [N>0] : m3 avec flèche PLEINE (synchrone) + message de retour pointillé ←
  • BREAK [erreur=true] : m4 dans le cadre BREAK. Si vrai, exécute m4 et abandonne tout le reste.
  • Création : m5 est une flèche allant vers un rectangle "Obj2:Logger" placé sur la ligne horizontale (pas en haut). La lifeline commence après la création.
Q6 4 pts Diagramme d'Activité — Gestion de Commande
Établissez le DAC du scénario suivant concernant le traitement d'une commande en ligne :
  • Le client passe une commande et choisit un mode de paiement
  • S'il paie par carte, le système vérifie la carte bancaire. Si refusée, la commande est annulée.
  • S'il paie par virement, un email de confirmation est envoyé
  • Une fois le paiement confirmé, la préparation du colis et la génération de la facture s'effectuent en même temps
  • Enfin, le colis est expédié au client
NB : Des points seront retirés pour chaque élément incorrect ou non demandé.
Barème : 4 × 0.5pt actions + 0.5pt nœud début + 0.5pt nœud fin + 0.5pt décision/union + 0.5pt bifurcation/jointure
Corrigé — Séquence :
  • ● Nœud initial
  • □ Passer commande
  • □ Choisir mode paiement
  • ◇ Décision : [Par carte] vs [Par virement]
  • Branche carte : □ Vérifier carte → ◇ Décision [Refusée → □ Annuler commande → ◎] [Acceptée → …]
  • Branche virement : □ Envoyer email confirmation → …
  • ◇ Fusion (merge) des deux branches vers ▬ Bifurcation (fork)
  • ▬ Fork → en parallèle : □ Préparer colis ET □ Générer facture
  • ▬ Join (jointure quand les deux sont prêts)
  • □ Expédier colis
  • ◎ Nœud final
🃏 Flashcards — Révision rapide
Clique sur une carte pour révéler la réponse
Différence Include vs Extend ?
Include = obligatoire, X inclut toujours Y (flèche de X → Y). Extend = optionnel/conditionnel, X peut étendre Y (flèche de X → Y, mais X est la variation).
Signe de l'attribut statique en UML ?
Attribut/méthode souligné. Correspond à static en Java.
Différence Agrégation vs Composition ?
Agrégation ◇ : vie indépendante, un objet peut appartenir à plusieurs agrégats. Composition ◆ : destruction du composite → destruction des composants, max 1 composite.
Quand utiliser une Classe Association ?
Quand une association * à * possède des attributs propres. En BD : table avec clé composée + attributs propres.
Différence Fork vs Join (DAC) ?
Fork (barre) : lance plusieurs flux en parallèle (1 entrée → N sorties). Join (barre) : synchronise, attend TOUS les flux (N entrées → 1 sortie).
Différence Merge vs Join (DAC) ?
Merge ◇ : accepte UN flot parmi plusieurs alternatifs. Join ▬ : attend TOUS les flots parallèles.
Fragment OPT vs ALT (DSE) ?
OPT : une seule condition, optionnel (si vrai → exécute). ALT : plusieurs conditions (si/sinon), séparées par tirets.
Fragment BREAK (DSE) ?
Si la condition est vraie : exécute le fragment ET abandonne tout le reste de l'interaction. Correspond à une interruption d'exception.
Message synchrone vs asynchrone (DSE) ?
Synchrone = flèche pleine ▶, l'expéditeur attend la réponse. Asynchrone = flèche ouverte ▷, l'expéditeur n'attend pas.
Convertir héritage acteurs en DCU sans héritage ?
Supprimer l'acteur parent. Connecter chaque acteur fils directement à TOUS les CUs (ceux du parent + les siens propres). Supprimer les CUs génériques si besoin.
Clé étrangère seule → relation DCL ?
Clé étrangère = Association (côté * là où se trouve la FK, côté 1 dans la table référencée).
11 qualités ISO/IEC 25010 ?
Validité, Fiabilité, Robustesse, Performance, Maintenabilité, Extensibilité, Réutilisabilité, Efficacité, Portabilité, Intégrité, Convivialité.
Acteur principal vs secondaire ?
Principal : déclenche les CUs, bénéficiaire direct. Secondaire : maintient/aide, pas bénéficiaire direct. Question clé : "Qui demande le service ?"
Axe Fonctionnel vs Statique vs Dynamique ?
Fonctionnel : ce que fait le système (DCU). Statique : comment il est structuré (DCL, DOB). Dynamique : comment il évolue (DAC, DSE).
Classe Abstraite en UML ?
Nom en italique OU stéréotype «abstract». Méthode abstraite : signature en italique ou «abstract». Impossible à instancier directement.
Nœud de fin d'activité vs fin de flot ?
Fin d'activité ◎ : arrête TOUS les flux. Fin de flot ⊗ : arrête UN flux seulement, les autres continuent.
🎯 Conseil exam : Dans le DCU, compter précisément les points attribués : 0.5pt par lien manquant/incorrect = perte rapide. Dessine d'abord le système (rectangle), puis les acteurs à l'extérieur, puis les CUs à l'intérieur, puis les relations.
